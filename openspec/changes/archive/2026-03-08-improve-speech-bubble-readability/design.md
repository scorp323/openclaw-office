## Context

当前 speech bubble 先显示为一个小型聊天图标，用户必须点击才会展开正文，且正文面板有固定宽高与较短的自动隐藏时间。这种交互对短句提示有效，但对 Markdown 长文、工具说明和演示场景不友好。PR #6 试图通过“大气泡默认可读 + 自适应 timeout + 手动关闭”改善体验，但其方案同时混入了 3D 放大渲染和额外视觉文案，风险偏大。

## Goals / Non-Goals

**Goals:**

- 让正在 speaking 的内容在 2D 视图中默认可读。
- 根据消息长度调整停留时间，并提供手动关闭能力。
- 优化气泡尺寸、滚动和边界定位，减少裁切。
- 将变更限制在可读性层面，避免引入新的演示/品牌逻辑。

**Non-Goals:**

- 不重做 3D 角色的整体展示策略。
- 不改变 Markdown 解析能力本身。
- 不引入与语言、品牌或调试模式相关的 UI 控件。

## Decisions

### 1. 采用“默认可读、可手动关闭”的 2D 气泡

在 active speaking 阶段，2D overlay SHALL 直接呈现可阅读正文，而不是仅给出展开图标。  
原因：阅读是主任务，展开按钮是额外阻力。

备选方案：

- 保持现状：对长文本不友好。
- 全面放大 2D/3D 两套气泡：收益高，但遮挡与复杂度也显著上升。

### 2. 停留时间与文本长度绑定

气泡 SHOULD 根据文本长度计算停留时间，并保留上下限；停止 speaking 后进入渐隐而不是立即消失。  
原因：长文本需要更长阅读时间，但无限停留会污染画面。

### 3. 先收敛 2D，再评估 3D

本 change 聚焦 `SpeechBubbleOverlay` 与 2D Office 体验；3D 只保持兼容，不强制同步为大气泡。  
原因：2D 是主路径，且 3D 叠加大气泡的遮挡和密度问题更难收敛。

### 4. 保证边界内展示

气泡位置与箭头对齐 SHOULD 根据左右边界自适应偏移，并允许正文区滚动。  
原因：长文本和边缘角色都容易触发裁切。

## Risks / Trade-offs

- [风险] 同时有多个 agent speaking 时，默认可读气泡会增加遮挡。  
  → Mitigation: 保持尺寸上限、允许滚动，并优先只优化 2D 主路径。

- [风险] 停留时间变长会让画面更“热闹”。  
  → Mitigation: 通过文本长度上下限和手动关闭按钮平衡密度。

- [风险] 旧测试依赖“点击图标后展开”行为。  
  → Mitigation: 将测试改为覆盖“默认可读 + 手动关闭 + 到时隐藏”的新契约。

## Migration Plan

1. 先调整 2D overlay 的展示策略和停留时间算法。
2. 再补手动关闭和边界定位逻辑。
3. 最后更新 UI 测试，验证 speaking、停止 speaking、长文本和边缘定位。
4. 如需回滚，可退回到旧的图标展开模式，不影响其它运行时逻辑。

## Open Questions

- 多个 agent 同时 speaking 时是否需要进一步的优先级或叠层策略。
- 3D 视图是否未来需要单独 change 来重做 speaking 展示。
