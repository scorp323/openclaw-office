# OpenClaw Office — 技术蓝图与架构设计

> 版本: v1.0 | 日期: 2026-02-25
> 状态: 设计阶段

---

## 一、项目定位

**OpenClaw Office** 是 OpenClaw Multi-Agent 系统的可视化监控前端。它将 Agent 的协作逻辑具象化为 isometric（等距投影）风格的"数字办公室"，让管理员能实时感知、理解和干预 Agent 的工作状态。

**核心隐喻：** Agent = 数字员工 → 办公室 = Agent 运行时 → 工位 = Session → 会议室 = 协作上下文

## 二、技术栈选型

### 2.1 选型原则
- 与 OpenClaw 主项目技术生态一致（TypeScript、ESM、现代前端）
- 渐进式复杂度：从 2D 可快速过渡到 2.5D/3D
- 社区成熟、文档充分、AI Coding 友好

### 2.2 最终选型

| 层 | 技术 | 版本/说明 | 选型理由 |
|-----|------|----------|---------|
| **构建工具** | Vite | 6.x | 与 OpenClaw 主项目一致，HMR 极快 |
| **UI 框架** | React | 19.x | 生态最大，R3F 依赖 React |
| **2D 渲染层** | SVG + CSS Animations | - | Phase 1 使用，零依赖，性能好 |
| **2.5D/3D 渲染层** | React Three Fiber (R3F) | 9.x | Three.js 的 React 封装，声明式 3D |
| **3D 辅助库** | @react-three/drei | latest | 提供 Html overlay、OrbitControls、环境光等常用组件 |
| **后处理** | @react-three/postprocessing | latest | Bloom、SSAO 等视觉特效 |
| **状态管理** | Zustand | 5.x | 轻量、高频更新友好、无 boilerplate |
| **样式** | Tailwind CSS | 4.x | 实用优先，快速构建 UI overlay |
| **动画** | Framer Motion | 12.x | 2D UI 动画；3D 动画由 R3F 自带处理 |
| **图表** | Recharts | 2.x | 轻量 React 图表库，用于 Token/Usage 面板 |
| **WebSocket** | 原生 WebSocket API | - | Gateway 使用标准 `ws` 协议，无需 Socket.io |
| **Markdown 渲染** | react-markdown | latest | Agent 输出内容渲染 |
| **包管理** | pnpm | workspace | 作为 OpenClaw monorepo 的 workspace app |
| **类型校验** | TypeScript | 5.x+ / tsgo | 严格模式，与主项目共享类型 |
| **代码规范** | Oxlint + Oxfmt | - | 与主项目一致 |

### 2.3 备选方案说明

| 考虑过的方案 | 未选择原因 |
|-------------|-----------|
| Unity WebGL | 打包体积大（30MB+），加载慢，与 TS 生态割裂 |
| Babylon.js | 社区和 React 集成不如 Three.js/R3F 成熟 |
| PixiJS (2D) | 缺少 3D 能力，后续升级需整体替换 |
| Socket.io | Gateway 使用原生 WebSocket，引入 Socket.io 增加不必要复杂度 |
| Redux Toolkit | 过于重量级，对高频 WS 事件不友好 |
| Jotai/Recoil | 对 3D 场景的全局状态管理不如 Zustand 直观 |

### 3.2 数据流

```
[Gateway Agent Runner]
    │
    ├─ emitAgentEvent({ stream: "lifecycle", data: { phase: "start" } })
    ├─ emitAgentEvent({ stream: "tool", data: { name: "browser_search" } })
    ├─ emitAgentEvent({ stream: "assistant", data: { text: "..." } })
    └─ emitAgentEvent({ stream: "lifecycle", data: { phase: "end" } })
            │
            ▼
[Gateway server-chat.ts — createAgentEventHandler()]
    │
    ├─ broadcast("agent", enrichedPayload)       → 所有 WS 客户端
    └─ broadcastToConnIds("agent", toolPayload)  → tool-events 订阅者
            │
            ▼
[Office Frontend — WebSocket Client]
    │
    ├─ onMessage → dispatch to Zustand store
    │   ├─ stream:"lifecycle" → 更新 agent.status (idle/working/error)
    │   ├─ stream:"tool"      → 更新 agent.currentTool + 弹出技能面板
    │   ├─ stream:"assistant" → 更新 agent.speechBubble (Markdown)
    │   └─ stream:"error"    → 更新 agent.status = "error" + 显示错误气泡
    │
    └─ Zustand store 变化 → React 自动重渲染对应组件
```

### 3.3 前端连接 Gateway 的认证流程

```
1. 前端 new WebSocket("ws://gateway:18789")
2. Gateway 发送 { type: "event", event: "connect.challenge", payload: { nonce } }
3. 前端发送 { type: "req", method: "connect", params: {
     minProtocol: 1, maxProtocol: 1,
     client: {
       id: "office-viz",
       version: "1.0.0",
       platform: "web",
       mode: "companion",      // 或新增 "visualization" 模式
     },
     caps: ["tool-events"],    // 订阅工具事件
     auth: { token: "<pairing-token>" }
   }}
4. Gateway 验证 → 发送 connect.accepted
5. 开始接收 broadcast 事件
```

### 4.1 Agent 状态机（基于实际 Gateway 事件）

```

                    │                                  │
                    ▼                                  │
     ┌──────┐   lifecycle    ┌──────────┐   lifecycle  │
     │ IDLE │──phase:start──>│ WORKING  │──phase:end───┘
     └──────┘               └──────────┘
         ▲                       │
         │                  ┌────┴────┐
         │                  ▼         ▼
         │            ┌──────────┐ ┌──────────┐
         │            │ THINKING │ │TOOL_CALL │
         │            │(无tool时) │ │(有tool时) │
         │            └──────────┘ └──────────┘
         │                  │         │
         │                  ▼         ▼
         │            ┌──────────────────┐
         │            │    SPEAKING      │
         │            │ (assistant流到达) │
         │            └──────────────────┘
         │                    │
         └────────────────────┘
```

### 4.2 Gateway 事件到可视化状态的映射表

| Gateway 事件 | stream | data 关键字段 | 前端 Agent 状态 | 视觉表现 |
|-------------|--------|-------------|----------------|---------|
| Agent 运行开始 | `lifecycle` | `phase: "start"` | `working` | 角色坐正，头顶出现加载动画 |
| LLM 正在思考 | `lifecycle` | `phase: "thinking"` | `thinking` | 头顶思考泡泡 + 闪烁效果 |
| 开始调用工具 | `tool` | `name: "browser_search"` | `tool_calling` | 面前弹出工具图标面板 |
| 工具返回结果 | `tool` | `result: {...}` | `working` | 工具面板显示完成勾选 |
| 输出文本 | `assistant` | `text: "..."` | `speaking` | 气泡显示 Markdown 内容 |
| 运行结束 | `lifecycle` | `phase: "end"` | `idle` | 角色回到休闲状态 |
| 运行出错 | `error` | `message: "..."` | `error` | 角色头顶红色叹号 |
| Sub-Agent 派生 | `lifecycle` | 自 subagent hook | `spawning` | 传送门特效 + 新角色出现 |
| Sub-Agent 结束 | `lifecycle` | 自 subagent hook | (移除) | 角色淡出消失 |

### 4.3 Zustand Store 设计

```typescript
// store/types.ts

type AgentVisualStatus =
  | "idle"
  | "thinking"
  | "tool_calling"
  | "speaking"
  | "spawning"
  | "error"
  | "offline";

type AgentPosition = { x: number; y: number; z: number };

type VisualAgent = {
  id: string;                    // agentId
  runId?: string;                // 当前 runId
  sessionKey?: string;           // 会话标识
  name: string;                  // 显示名称
  status: AgentVisualStatus;
  position: AgentPosition;       // 3D 坐标
  zone: "desk" | "meeting" | "hotdesk" | "lounge";

  // 实时内容
  currentTool?: { name: string; args?: Record<string, unknown> };
  speechBubble?: { text: string; timestamp: number };

  // Sub-Agent 关系
  isSubAgent: boolean;
  parentAgentId?: string;
  childAgentIds: string[];

  // 统计
  tokenUsage?: { input: number; output: number; total: number };
  toolCallCount: number;
  lastActiveAt: number;

  // 外观
  avatarSeed: string;            // 用于生成确定性的 Avatar 外观
  color: string;                 // 主题色
};

type CollaborationLink = {
  fromAgentId: string;
  toAgentId: string;
  strength: number;              // 0-1，用于控制连线亮度/粗细
  lastMessageAt: number;
};

type OfficeStore = {
  // 数据
  agents: Map<string, VisualAgent>;
  links: CollaborationLink[];
  globalMetrics: {
    totalTokens: number;
    activeAgents: number;
    collaborationHeat: number;
    tokenRate: number;            // tokens/sec
  };

  // 连接状态
  connectionStatus: "connecting" | "connected" | "disconnected" | "error";
  lastEventSeq: number;

  // UI 状态
  selectedAgentId: string | null;
  viewMode: "overview" | "meeting" | "detail";
  renderMode: "2d" | "isometric" | "3d";
  theme: "light" | "dark";

  // Actions
  processAgentEvent: (event: AgentEventPayload) => void;
  selectAgent: (id: string | null) => void;
  setViewMode: (mode: "overview" | "meeting" | "detail") => void;
};
```

## 六、关键技术方案

### 6.1 WebSocket 客户端（gateway/ws-client.ts）

**核心职责：**
- 连接 Gateway 并完成 challenge + connect 认证
- 自动重连（指数退避：1s → 2s → 4s → ... → 30s max）
- 事件分发到 Zustand store
- 心跳保活

**关键行为：**
```typescript
// 伪代码示意
class GatewayWsClient {
  connect(url: string, authToken: string): void;
  // 收到 connect.challenge → 发送 connect request
  // 收到 connect.accepted → 标记已连接
  // 收到 event:agent → store.processAgentEvent(payload)
  // 收到 event:presence → store.updatePresence(payload)
  // 收到 event:health → store.updateHealth(payload)

  request<T>(method: string, params?: unknown): Promise<T>;
  // 用于 RPC 调用如 agents.list, usage.status
}
```

### 6.2 事件节流策略

当多个 Agent 并发运行时，WebSocket 事件可能达到 50-100+ 条/秒。

**策略：**
1. **批量更新**：每 100ms 收集事件，批量更新 Zustand store（使用 `unstable_batchedUpdates`）
2. **优先级队列**：`lifecycle` 和 `error` 事件立即处理；`assistant` 文本流可做 150ms 节流；`tool` 事件正常处理
3. **视口剪裁**：仅更新当前视口内可见 Agent 的渲染属性（3D 模式下）

### 6.3 工位分配算法（position-allocator.ts）

Agent 在办公室中的位置不由 Gateway 决定，前端自行分配：

```
1. 常驻 Agent → 固定工位区（按 agentId 的 hash 确定性分配到预设位置）
2. Sub-Agent → 热工位区（动态分配到最近的空闲位置）
3. 协作中的 Agent → 当协作链接数 >= 2 时，自动移入会议区
4. 空闲 Agent → 可随机分配到"休息区"（咖啡区等，增加拟人感）
```

### 6.4 Avatar 生成（avatar-generator.ts）

基于 `agentId` 的确定性 hash 生成外观特征：

- 颜色方案（头发/衣服/肤色）
- 几何体形状（Phase 1 用简单几何体）
- 3D 模型变体（Phase 2 用 GLTF 模型的材质变化）

这确保同一个 Agent 在每次打开页面时看起来一样。

### 6.5 渲染模式切换

支持三种渲染模式，通过 URL 参数或 UI 切换：

| 模式 | 技术 | 适用场景 |
|------|------|---------|
| `2d` | SVG + CSS | 快速概览、低端设备、大量 Agent |
| `isometric` | CSS 3D transform / R3F (低多边形) | 默认视图，平衡美观与性能 |
| `3d` | R3F + postprocessing | 展示模式、少量 Agent 深度交互 |

## 七、Gateway 端适配需求

### 7.1 无需改动（直接可用）

- Agent 生命周期事件（`lifecycle` stream）
- 工具调用事件（`tool` stream）
- 文本输出事件（`assistant` stream）
- RPC：`agents.list`, `sessions.list`, `usage.status`, `tools.catalog`

### 7.2 建议增加（可选优化）

| 改动 | 说明 | 优先级 |
|------|------|--------|
| `visualization` 客户端 capability | 允许可视化客户端接收全量事件而不被限流 | P1 |
| Sub-Agent 树广播 | 在 `subagent_spawned` 事件中通过 broadcast 推送父子关系 | P1 |
| `agents.status` RPC 方法 | 返回所有 Agent 当前运行状态的快照（用于初始化/重连） | P2 |
| Token usage 流式推送 | 在 `lifecycle` phase:end 事件中附带本次 run 的 token 消耗 | P2 |
| `agent.force` RPC 方法 | 允许前端向指定 Agent 发送干预指令 | P3 |

### 7.3 集成方式选择

**推荐：作为 OpenClaw Extension（插件）**

```
extensions/office-viz/
├── openclaw.plugin.json
├── package.json
├── src/
│   └── index.ts              # 注册 HTTP 路由 + 额外 RPC 方法
└── dist/                      # 前端构建产物
    ├── index.html
    ├── assets/
    └── ...
```

插件入口利用 `registerHttpRoute` 提供静态前端资源：

```typescript
export const plugin: OpenClawPluginDefinition = {
  id: "office-viz",
  name: "OpenClaw Office Visualizer",
  register(api) {
    // 提供前端静态资源
    api.registerHttpRoute({
      path: "/office",
      handler: serveStaticFromDist,
    });

    // 注册可视化专用 RPC（可选）
    api.registerGatewayMethod("viz.agents.status", async (req) => {
      // 返回所有 Agent 的当前可视化状态快照
    });
  },
};
```

**好处：**
- 无需独立部署，Gateway 启动即可访问 `http://gateway:18789/office`

- 可通过 plugin hooks 获取更丰富的数据

## 八、性能预算

| 指标 | 目标值 | 措施 |
|------|--------|------|
| 首屏加载时间 | < 3s (LCP) | Vite 代码分割 + R3F 懒加载 |
| 事件处理延迟 | < 200ms（从 Gateway 发出到视觉更新） | 事件批处理 + requestAnimationFrame |
| 内存占用 | < 200MB（50 Agent 场景） | InstancedMesh + 对象池 |
| 帧率 | ≥ 30fps（2D）/ ≥ 24fps（3D） | LOD 系统 + 视口剪裁 |
| WebSocket 带宽 | < 100KB/s（正常工作负载） | 仅订阅必要 capability |
| 最大支持 Agent 数 | 100+（2D）/ 50（3D） | 超阈值自动切 2D 模式 |

## 九、安全与权限

1. **认证**：复用 Gateway 的 pairing 机制。首次访问需输入 pairing code
2. **权限**：Force Action 操作需要 `operator` scope 权限
3. **数据安全**：前端不存储敏感数据（token、session 内容仅在内存中，刷新即清）
4. **CORS**：前端与 Gateway 同源（同一 HTTP 服务），无 CORS 问题
