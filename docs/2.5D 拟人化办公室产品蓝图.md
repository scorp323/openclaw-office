# 1. 产品定义

## 1.1 产品名称

**OpenClaw Living Office**
副标题：**把 Agent Runtime 变成可感知的公司运转沙盘**

## 1.2 产品目标

让管理员在 3–8 秒内看懂：

* 现在有没有新事务进入
* 是谁在主导处理
* 是否调用了工具或外部系统
* 是否拉起 sub-agent 协作
* 目前是执行中、等待中还是阻塞中
* heartbeat / cron 是否在正常运作
* 结果有没有回到主线

## 1.3 非目标

这版不要追求：

* 完整替代日志与审计系统
* 完整替代 OpenClaw Web/CLI 控台
* 过度游戏化
* 所有人物都“持续思考、持续乱动”

# 2. 第一性原理

## 2.1 根问题

机器处理速度是毫秒级。
人的理解速度是秒级。
如果你把真实 runtime 直接渲染成动画，用户会看到：

* 瞬移
* 闪烁
* 不明所以的状态切换
* 很多动作，但没有因果

所以必须建立一层：

## 2.2 核心架构

**Runtime Truth Layer → Perception Layer → 2.5D Scene Layer**

### Runtime Truth Layer

只接收真实事件：

* gateway event
* agent state
* session state
* heartbeat
* cron
* health
* tool call
* sub-agent lifecycle
* error / timeout / blocked

### Perception Layer

做事件压缩、节流、聚合、叙事化。

### 2.5D Scene Layer

把叙事化后的状态映射成：

* 小人动作
* 区域灯光
* 广播牌
* 白板内容
* 路径线
* 状态停留

# 3. 世界观映射

## 3.1 真实机制 → 公司隐喻

| OpenClaw 机制     | 办公室隐喻       | 设计说明                |
| --------------- | ----------- | ------------------- |
| Gateway         | 总机中控大厅      | 不是门，而是前台+总机+调度+事件总线 |
| Agent           | 正式员工        | 常驻角色，有岗位和职责         |
| Sub-agent       | 临时工/项目协作者   | 只在项目室亮起，不占正式工位      |
| Chat/Session    | 客户来访/工单/会话  | 是事务载体，不是角色          |
| Presence        | 在岗灯/状态牌     | 表示是否在线、可响应          |
| Health          | 健康看板/楼宇状态   | 属于系统层，不属于某个人        |
| Heartbeat       | 巡检机制        | 用灯光/扫描波表示，不拟人       |
| Cron            | 行政广播/制度任务   | 用广播屏和时钟机制表示         |
| Tool call       | 去设备间/系统终端办事 | 不要一律走路，用状态图标更准      |
| Queue           | 待办池/前台待分发   | 在 Gateway 区显示       |
| Blocked/Timeout | 卡点/堵塞/等待外部  | 必须显性停留              |

# 4. 产品体验原则

## 4.1 状态优先，动作次之

用户首先要看懂“发生了什么”，其次才是“怎么动”。

## 4.2 走路是昂贵语言

只有这几类事件值得让角色移动：

* 接到主线新任务
* 去白板讨论
* 进入项目室拉起协作
* 回传结果到主线
* 从阻塞切到排障

其他大量事件不该靠走路表达。

## 4.3 高频事件不做高频动画

heartbeat、presence、health 这类高频系统事件，优先用：

* 呼吸灯
* 工位状态圈
* 局部扫描波
* 仪表牌变化

不要让角色来回跑。

## 4.4 异常要比正常更慢

正常完成可以快。
阻塞/失败/重试必须停留更久，让人能看见。

## 4.5 主线永远清晰

无论多复杂，用户都能回答：

“现在谁在牵头处理这件事？”

# 6. 场景规划

## 6.1 空间分区

### 1）Gateway 中控大厅

职责：

* 外部事务进入
* 队列分发
* 广播 heartbeat / cron / health 机制状态
* 显示系统主路由

视觉：

* 前台总机台
* 进件传送带
* 调度屏
* 闸机 / 光带 / 分发箭头

### 2）正式员工工区

职责：

* 正式 agent 日常处理主线任务
* 明确角色分工

视觉：

* 每人一个工位
* 工位上有 role 标签
* 桌面设备图标随任务变化
* 顶部状态灯颜色统一

### 3）共享白板区

职责：

* 展示当前目标、上下文摘要、共享事实
* 显示主线任务的阶段性结论

视觉：

* 白板/大屏
* 仅保留高价值摘要
* 不直接倒日志

### 4）项目室 / 临时协作区

职责：

* sub-agent 出现与消失
* 临时协作任务
* 多 agent 联动时的可视聚焦

视觉：

* 玻璃会议室
* 临时亮灯
* 完成后熄灭
* 可出现 1–3 个临时工位或影子角色

### 5）系统设施区

职责：

* cron 广播
* heartbeat 巡检
* health 监测

视觉：

* 广播钟
* 巡检波纹
* 服务健康塔
* 系统灯墙

### 6）休息与缓冲区

职责：

* 让场景更像办公室
* 承接无任务时的轻 idle

视觉：

* 饮水区
* 沙发
* 植物
* 走廊

注意：这里只做弱动画，不能喧宾夺主。

* 在项目室生成

* 可视状态更轻

# 8. 状态机设计

## 8.1 Agent 可视状态机

```text
IDLE
→ INCOMING
→ ACK
→ WORKING
→ TOOL_CALL
→ WAITING
→ COLLABORATING
→ RETURNING
→ DONE

异常支线：
WORKING / TOOL_CALL / WAITING
→ BLOCKED
→ RETRYING
→ RECOVERED
```

## 8.2 每个状态的视觉规则

| 状态            | 小人动作       | 工位变化    | HUD 变化 | 停留建议     |
| ------------- | ---------- | ------- | ------ | -------- |
| IDLE          | 轻呼吸/轻摆头    | 绿灯      | 无      | 持续       |
| INCOMING      | 看向 Gateway | 工位微亮    | 新事务提示  | 0.8–1.2s |
| ACK           | 短移动或点头     | 黄灯      | “已接单”  | 1–1.5s   |
| WORKING       | 坐在工位/专注    | 黄灯+桌面图标 | 当前任务摘要 | 2–5s     |
| TOOL_CALL     | 不一定移动      | 工具图标高亮  | 显示工具名  | 1.5–4s   |
| WAITING       | 停滞/轻闪烁     | 黄灯变慢闪   | “等待返回” | 2–6s     |
| COLLABORATING | 去白板/项目室    | 多工位联动   | 协作链可见  | 4–10s    |
| RETURNING     | 回主线        | 路径回传    | “回传结果” | 1–2s     |
| DONE          | 短完成反馈      | 绿灯恢复    | “已完成”  | 1–1.8s   |
| BLOCKED       | 静止+红提示     | 红灯      | 异常说明   | 4–8s     |
| RECOVERED     | 从红转黄再绿     | 恢复动画    | “已恢复”  | 2–3s     |

# 9. 感知节流规则

这是最关键的开发规则。

## 9.1 事件分级

### L0：毫秒系统噪声

例如高频 presence、内部确认、重复健康信号。
处理：只更新内部状态，不上屏。

### L1：轻事件

如 heartbeat poll、简短 tool ack。
处理：只改工位灯、状态圈、图标。

### L2：短任务

如接单、短分析、写入上下文。
处理：工位层动作，不跨区移动。

### L3：中任务

如协作、工具调用等待、排障。
处理：允许跨区移动。

### L4：重要事件

如新客户进入、失败重试、cron 主任务、sub-agent 拉起。
处理：强叙事、保留更久、可写日志。

## 9.2 时间策略

| 类型    | 最短展示 |
| ----- | ---- |
| 状态闪变  | 0.8s |
| 短工位动作 | 1.5s |
| 主线处理  | 2.5s |
| 协作场景  | 4s   |
| 阻塞异常  | 6s   |
| 重大恢复  | 3s   |

## 9.3 聚合策略

300–800ms 内连续到来的关联事件，聚合为一个叙事片段。
例如：

```text
client.message
session.open
presence.sync
agent.route
```

不要渲染四次。
渲染成：

**“客户消息到达，Gateway 完成分发，Sales Agent 开始接单。”**

# 10. 事件到动画的映射语法

## 10.1 统一行为 DSL

```text
ARRIVE
DISPATCH
ACK
FOCUS
CALL_TOOL
WAIT
SPAWN_SUBAGENT
COLLAB
RETURN
BROADCAST_CRON
POLL_HEARTBEAT
BLOCK
RECOVER
```

## 10.2 映射范式

### ARRIVE

* Gateway 传送带亮起
* 前台出现事务卡
* 一条路径线指向 GM 或目标 agent

### DISPATCH

* Gateway 发出光带
* 目标工位高亮
* 目标 agent 看向前台

### POLL_HEARTBEAT

* 楼层巡检波扫过
* 工位圈短亮
* 不触发角色走动

### BROADCAST_CRON

* 广播牌亮
* 行政钟扩散波纹
* 对应工位收到制度任务

### SPAWN_SUBAGENT

* 项目室亮灯
* 临时工位升起
* 临时角色淡入

### BLOCK

* 路径中断
* 目标工位红灯
* 日志写入“等待外部返回/超时”

### RECOVER

* IT 或主负责人介入
* 红转黄再转绿
* 日志写入恢复结论

# 13. 前端技术蓝图

## 13.1 技术建议

你现在最合适的路线：

* **React + TypeScript**
* **PixiJS 或 Phaser** 做 2.5D 场景渲染
* DOM 只负责 HUD / 日志 / 操作台
* Zustand 或 Jotai 管状态
* Framer Motion 仅做 HUD 微动效
* 场景数据和动画状态解耦

## 13.2 为什么不建议纯 DOM/CSS

纯 HTML/CSS 做原型可以。
但要做成演示级产品，会遇到：

* 复杂路径动画难维护
* 多角色并行动画混乱
* 透视、层级、碰撞和视口切换难做
* 帧率控制差

所以：

### V1 原型

HTML/CSS/JS

### V2 可演示

React + PixiJS

### V3 可生产

React + PixiJS + Runtime Adapter + Replay System

# 14. 后端 / Runtime Adapter 蓝图

## 14.1 职责

Adapter 不修改 OpenClaw，而是订阅 Gateway 事件并投影。

OpenClaw Gateway 当前通过 WS 暴露 typed API 并推送多种 server-side events，因此最稳妥的方式是做一个旁路 adapter，把原始事件转成 projection event，再喂给前端。([GitHub][1])

## 14.2 Adapter 模块

### A. Gateway Connector

* 连接 OpenClaw Gateway WS
* 订阅 agent/chat/presence/health/heartbeat/cron

### B. Event Normalizer

* 统一 event schema
* 补 session/agent/task 关联键

### C. Perception Engine

* 聚合
* 节流
* 分级
* 叙事化
* 设定 hold 时间

### D. Projection Store

* 保存当前可视状态
* 提供 replay
* 支持前端断线重连

### E. Stream API

* SSE 或 WS 推送给前端

# 16. 美术与动效规范

## 16.1 风格

不是卡通儿童风。
建议：

* 深色科技办公室
* 轻赛博，但不能过度霓虹
* 2.5D 等距视角
* 角色拟人，但不过分 Q 版
* 重点在“组织活性”，不是“角色卖萌”

## 16.2 动效准则

* idle 动作小
* 工作动作克制
* 协作动作明确
* 异常动作显性
* 心跳动作设施化
* cron 动作制度化

## 16.3 镜头策略

V1 固定镜头。
V2 加轻微聚焦：

* 新任务时轻推向 Gateway
* 协作时轻推项目室
* 异常时轻推阻塞工位

不要上来做大幅镜头飞来飞去。

# 18. 验收标准

这个产品是否成功，不看“酷不酷”，看这 8 条：

1. 用户第一次看 5 秒内能知道谁在主导
2. heartbeat 和 cron 看起来像机制，不像某个员工技能
3. 高频事件不会导致小人乱跑
4. sub-agent 不会污染正式工区
5. 异常一定能被看见
6. 白板展示的是上下文，不是垃圾日志
7. 主舞台永远不被 HUD 遮住
8. 接真实 Gateway 后，不会因为事件频繁而视觉崩坏
